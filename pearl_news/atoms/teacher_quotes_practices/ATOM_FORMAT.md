# Teacher Atom File Format

Each topic gets a `topic_{topic}.yaml` file in this directory.

## What is an atom?

An atom is a single, approved teaching statement from a specific teacher. It is:

- **Sourced**: derived from the teacher's actual published teachings, talks, or writings
- **Specific**: states one clear idea, not a vague platitude
- **Connectable**: can be connected to a news event (it's not abstract philosophy)
- **30-50 words**: long enough to carry meaning, short enough for the LLM to embed in a paragraph

## What makes a GOOD atom?

Good: "When communities face disruption, the first task is not to react but to notice what is actually happening — to distinguish between the story the mind tells and the reality the body perceives."

Bad: "Peace is important and we should all strive for inner harmony." (platitude, not connectable to news)

Bad: "Be mindful." (too short, too generic)

## File structure

```yaml
topic_key: climate          # matches the topic classifier output
description: "Teacher atoms for climate-related articles"

teachers:
  ahjan:                # must match teacher_id in teacher_news_roster.yaml
    display_name: "Ajahn Ahjan"
    tradition: "Theravada Buddhist"
    attribution: "From within the Theravada Buddhist tradition, Ajahn Ahjan teaches that"
    status: starter     # Atom lifecycle: starter | reviewed | approved
    atoms:
      - > first atom (40-80 words, tradition-specific, news-connectable)
      - > second atom (different concept from same tradition)
      # ... 10 atoms per teacher per topic
```

## Atom lifecycle states

Every teacher entry in a topic file has a `status` field tracking where its atoms are in the review process:

**STARTER** — Auto-generated by LLM scaffolding (`teacher_onboarding.py --scaffold`). Not used in production. Used in dev/test for rapid iteration. Requires human editorial review before advancing.

**REVIEWED** — Human editor has verified the atoms against the teacher's actual published teachings. Checked for tradition-specificity, news-connectability, and word count. Still not in production; marked for final approval.

**APPROVED** — Cleared for production use. Appears in published articles. The teacher resolver filters to APPROVED atoms only when `PEARL_NEWS_ENV=production`.

Status upgrade process:
```
STARTER (generated) → REVIEWED (editor verified) → APPROVED (production-ready)
```

Use `teacher_onboarding.py --fix-status teacher_id topic new_status` to batch-update after review.

## Populating atoms

**Manual process:**
1. Read the teacher's doctrine files in `SOURCE_OF_TRUTH/teacher_banks/`
2. Extract 10 teachings per teacher that connect to this topic
3. Write each as a 40-80 word statement that a journalist could embed in a news article
4. Review: could this atom be connected to a real news event? If not, rewrite.

**Automated scaffolding:**
```bash
# Audit gaps
python -m pearl_news.pipeline.teacher_onboarding --audit

# Generate missing atoms via LLM (dry-run first)
python -m pearl_news.pipeline.teacher_onboarding --scaffold --dry-run
python -m pearl_news.pipeline.teacher_onboarding --scaffold

# Validate all teachers are complete
python -m pearl_news.pipeline.teacher_onboarding --validate
```

All scaffolded atoms start as `status: starter` — never auto-approved.
